# Background sync reliability on macOS and Linux

This is a pattern survey for a turnkey CLI that archives append-only, still-growing session files. It catalogs mechanisms and their failure boundaries; it does not choose Packbat's design. Sources are operating-system documentation, project documentation, and project source repositories.

## Baseline: the dogfood implementation

`~/bank/personal/bin/archive-sessions` is a stateless, idempotent sweep over three source trees: Claude projects, active Codex sessions, and archived Codex sessions. It finds `*.jsonl`, mirrors each relative path under `~/session-archive/<lowercase-short-hostname>/<harness>/`, and compresses it with a fixed Homebrew `zstd` path. Existing output is overwritten only when the source mtime is newer. It never deletes, never merges machines, and deliberately has no off-box transfer. Every run regenerates a newest-first `index.tsv` from archived-file mtimes and appends a completion line to `~/Library/Logs/session-archive.log`. Missing source trees are success; compression or `mkdir` failures are not reliably surfaced because the script has `set -u`, but not `set -e` or explicit status aggregation.

`~/Library/LaunchAgents/com.liam.session-archive.plist` is a per-login-user launchd agent. It invokes the script through `/bin/sh`, has `RunAtLoad=true`, schedules 21:00 daily with `StartCalendarInterval`, and appends stderr to the same unrotated log as the script's stdout-like completion record. This is the **scheduled full sweep** pattern: low resident cost and naturally repairs missed file-level events because every run re-enumerates current state. Calendar scheduling catches a time missed while sleeping on wake, but not one missed while powered off; `RunAtLoad` partly covers reboot/login. Apple documents both distinctions explicitly ([Apple, Scheduling Timed Jobs](https://developer.apple.com/library/archive/documentation/MacOSX/Conceptual/BPSystemStartup/Chapters/ScheduledJobs.html)). The plist has no explicit overlap lock, success stamp, failure backoff policy, or archive-to-source reconciliation, so its log can say a run completed even after some commands failed.

## Pattern catalog

### 1. Wizard-written scheduler units

#### launchd user agents

A wizard can write a unique plist under `~/Library/LaunchAgents`, validate it with `plutil`, and bootstrap it into `gui/$UID` with `launchctl bootstrap`; uninstall reverses that with `bootout` and removes only the wizard-owned file. Use absolute executable and state paths because a launch agent gets a small, non-shell environment.

`StartInterval=N` means a relative interval trigger. Apple says interval jobs are skipped while the machine is asleep or off and wait for the *next* interval. `StartCalendarInterval` means calendar fields; if the scheduled instant passes during sleep, launchd runs one job on wake, while an instant missed during power-off is skipped until the next designated time. Multiple calendar instants missed during one sleep are coalesced into one wake run. This makes calendar scheduling materially better for laptop catch-up than interval scheduling, although modern sleep/DarkWake details can vary; the contractual behavior is the archived Apple documentation ([Apple, Scheduling Timed Jobs](https://developer.apple.com/library/archive/documentation/MacOSX/Conceptual/BPSystemStartup/Chapters/ScheduledJobs.html), [Apple, Creating Launch Daemons and Agents](https://developer.apple.com/library/archive/documentation/MacOSX/Conceptual/BPSystemStartup/Chapters/CreatingLaunchdJobs.html)). `RunAtLoad` adds a login/bootstrap sweep but is not a historical replay.

launchd does not launch a second copy while a job is already running, which prevents scheduler-originated overlap. Its default throttling discourages rapid crash loops, and `ThrottleInterval` can tune that; `KeepAlive` is appropriate for a daemon, not a periodic one-shot ([Apple launchd source and manual](https://github.com/apple-oss-distributions/launchd/blob/main/man/launchd.plist.5)). A process started manually outside launchd can still overlap, so a job-level lock remains useful when `doctor`, hooks, or manual runs share the same sync entry point.

Strengths: native, unprivileged, no resident Packbat process, calendar wake catch-up, automatic serialization, inspectable registration and exit state. Weaknesses: login-domain lifetime, no power-off catch-up, no record of every intended firing, fragile permissions/privacy access to agent-session directories, and plain `StandardOutPath`/`StandardErrorPath` files do not rotate themselves.

#### systemd user service plus timer

A wizard can write `~/.config/systemd/user/packbat-sync.service` and `.timer`, then run `systemctl --user daemon-reload` and `systemctl --user enable --now packbat-sync.timer`. A oneshot service separates execution policy from the timer. `OnCalendar=` expresses wall-clock schedules. `Persistent=true` stores the last trigger time; when an inactive timer becomes active, systemd immediately triggers if at least one calendar event was missed. It catches up **once**, not once per missed occurrence. `Persistent` affects `OnCalendar` timers, not monotonic `OnActiveSec`/`OnUnitActiveSec` timers ([systemd.timer](https://www.freedesktop.org/software/systemd/man/latest/systemd.timer.html)).

The user manager normally exists with the user's login session. `loginctl enable-linger "$USER"` makes it start at boot and remain after logout, allowing user services to run without an active login ([loginctl](https://www.freedesktop.org/software/systemd/man/latest/loginctl.html)). Lingering changes host-wide user-manager lifecycle and may be disabled by policy, so a wizard should explain and separately verify it. Without linger, `Persistent=true` catches a missed calendar time when the user next logs in, provided the timer was enabled and its timestamp survived; it cannot execute while no user manager exists.

If the triggered service is already active, systemd does not restart another instance, so timer-originated overlap is avoided ([systemd.timer](https://www.freedesktop.org/software/systemd/man/latest/systemd.timer.html)). `RandomizedDelaySec` can avoid synchronized load; `AccuracySec` permits coalescing for power efficiency. Service-level `Restart=` can respawn a daemon, but is generally not useful for a failed oneshot; let the next timer run or add bounded application retry/backoff. The journal supplies bounded/rotated logging under distro policy, unlike an unmanaged file ([journald.conf](https://www.freedesktop.org/software/systemd/man/latest/journald.conf.html)).

Strengths: explicit durable catch-up, rich machine-readable state, lifecycle and log management. Weaknesses: systemd is not universal across Linux, user-manager/linger subtleties, only one catch-up activation, and a timer timestamp proves triggering rather than successful coverage.

#### cron fallback

A wizard can preserve the existing crontab, add a uniquely delimited line via `crontab`, and use absolute paths plus explicit `PATH`, `HOME`, and locale. Cron evaluates calendar matches while its daemon and machine are running; ordinary cron has no sleep/off catch-up. Apple explicitly says missed cron jobs wait for the next designated time, and also recommends launchd instead on macOS ([Apple, Scheduling Timed Jobs](https://developer.apple.com/library/archive/documentation/MacOSX/Conceptual/BPSystemStartup/Chapters/ScheduledJobs.html)). Cron starts commands with a minimal/default environment, normally `/bin/sh`; PATH, shell startup files, working directory, `%` escaping, secrets, and mail-based error reporting are recurring pitfalls ([crontab(5)](https://man7.org/linux/man-pages/man5/crontab.5.html)). Cron also has no portable per-job last-run or exit-status database and does not prevent overlap.

Strengths: broad availability and simple fallback. Weaknesses: no native catch-up, weak introspection, shared crontab editing risk, environment drift, no lifecycle/log policy, and no overlap control. A stamp written atomically by the job plus a sweep that uses durable source/archive state is mandatory if `doctor` must say more than “the line exists.”

#### Installer precedents

- **resticprofile** is the closest schedule-install wizard precedent. `schedule`, `unschedule`, and `status` create/read OS-native schedules. Its systemd path writes service/timer units, reloads, enables, and starts them; it supports system, system-as-user, and logged-on user scopes, and documents linger for the last case ([schedule commands](https://creativeprojects.github.io/resticprofile/schedules/commands/), [systemd backend](https://creativeprojects.github.io/resticprofile/schedules/systemd/)). Its macOS backend creates launchd schedules, and its scheduler abstraction exposes platform-specific status rather than pretending all platforms have identical evidence.
- **autorestic** puts cron expressions in its own config but still needs an external frequent call to `autorestic cron`; its docs tell the user to edit crontab and recommend an invocation every five minutes with explicit `PATH` and `--ci`. It does not install the OS schedule. This is a two-level scheduler and polling dispatcher ([autorestic cron docs](https://git.nicco.io/cupcakearmy/autorestic/src/commit/fc2756124e74ac20a2b4fe3dc2545f329a43f28c/docs/markdown/location/cron.md)).
- **borgmatic** deliberately delegates scheduling. Its official setup offers sample cron or systemd files and `systemctl enable --now`, rather than a cross-OS install command; file logging requires the user to configure rotation ([borgmatic setup](https://torsion.org/borgmatic/how-to/set-up-backups/), [borgmatic logging](https://torsion.org/borgmatic/reference/command-line/logging/)). This keeps policy visible but is less turnkey.

### 2. Long-lived daemon plus filesystem watch

A daemon watches source roots and nudges a sync when files appear or change. It can deliver seconds-level latency and amortize directory discovery, but notifications are hints, not a completeness proof.

On macOS, FSEvents watches directory hierarchies efficiently and its on-volume event journal can resume from a stored event ID across daemon downtime and reboot. Events may be coalesced at directory granularity. `MustScanSubDirs`, `UserDropped`, or `KernelDropped` require a recursive rescan; root moves need `WatchRoot`. Apple recommends starting the stream before the initial scan and rescanning directories changed during that scan to close the setup race ([Apple FSEvents guide](https://developer.apple.com/library/archive/documentation/Darwin/Conceptual/FSEvents_ProgGuide/UsingtheFSEventsFramework/UsingtheFSEventsFramework.html)). Persistent IDs improve recovery but can become invalid after journal purge, volume replacement/restore, or ID rollback; a full sweep remains the final repair path.

On Linux, inotify is not recursive: the daemon adds one watch per directory and must install watches for newly created directories. Per-user `max_user_watches`, `max_user_instances`, and `max_queued_events` limit coverage; queue overflow emits `IN_Q_OVERFLOW`, after which state must be rescanned. Recursive setup itself has a race while watches are being installed ([inotify(7)](https://man7.org/linux/man-pages/man7/inotify.7.html), [inotifywait(1)](https://man7.org/linux/man-pages/man1/inotifywait.1.html)). Network and unusual filesystems may not provide useful native notifications.

Debouncing should be trailing-edge with a maximum wait: accumulate paths for a short quiet window, but force a sweep after a bounded interval while a JSONL file is continuously appended. Sync should read/copy a consistent observed version and allow the next event/sweep to refresh a growing session. A dirty flag set while sync is running produces exactly one follow-up run rather than parallel runs. The durable checkpoint is not “event received”; it is the archive cursor/content state committed after output is durable.

Use one process owner per user. launchd `KeepAlive=true` or a systemd user service with `Restart=on-failure` and bounded `RestartSec`/`StartLimit*` supplies respawn and backoff. Both managers avoid duplicate manager-launched instances; an application lock still unifies manager, hook, manual, and `doctor --repair` entry points. On Linux, linger controls boot/no-login availability. On macOS, a LaunchAgent starts in the GUI login domain, so it is not a pre-login daemon.

Blind spots: events while the watcher is dead, dropped/coalesced queues, watch-limit exhaustion, directories added during recursive setup, inaccessible roots, replaced roots/volumes, and a live-but-stuck event loop. A periodic full sweep plus heartbeat/progress stamps is what makes this architecture auditable.

### 3. Hook-based triggers

The harness can invoke a fast `packbat sync --nudge` itself. Claude Code exposes `Stop` when the main agent finishes responding and `SessionEnd` when a session ends; `Stop` can repeat during an interactive session, while `SessionEnd` is semantically closer to finalization. Hooks are commands configured in Claude settings and receive structured input ([Anthropic hook reference](https://code.claude.com/docs/en/hooks)). A hook should enqueue or request a sweep and return quickly rather than compress synchronously on the harness's critical path.

Shell `precmd` runs before each prompt and `chpwd` after directory changes in shells that provide them. They are frequent nudges but say nothing about writes performed by GUI apps, noninteractive shells, other harnesses, or a terminal killed without returning to a prompt. Git hooks are a useful analogy: event-local, repository-local, and not cloned by default. Git documents that hooks live in the local hooks directory and that a missing executable hook is ignored ([githooks](https://git-scm.com/docs/githooks)).

Hooks have the lowest normal latency and no resident watcher cost, but the narrowest completeness boundary. A process can be killed, crash, lose power, or sleep before/while its end hook runs; hooks can be disabled or overwritten; new harness versions/providers need integration; and session files may still be flushing when the hook fires. Therefore hook success cannot prove all source files are archived. It is best modeled as an on-event nudge to an idempotent sweep, backed by scheduled reconciliation.

### 4. Hybrids and anacron-style catch-up

**Scheduled sweep plus on-event nudge** combines low latency with bounded recovery. Hooks or watchers request the same serialized sweep that the OS timer runs. The timer repairs lost nudges, dropped events, watcher downtime, and growing files. The sweep updates one durable success/coverage record, so all trigger paths share semantics.

**Watcher plus scheduled sweep** is Syncthing's shape: native events accelerate discovery, while a periodic rescan catches notification loss. This costs a resident process and occasional O(tree) scans but turns watcher failure from permanent loss into bounded delay.

**Anacron-style dispatcher** records the last successful period and checks “is work overdue?” at login, wake/startup, and periodic ticks. Traditional anacron is intended for machines not running continuously and uses timestamp files to run daily-or-longer jobs after downtime ([anacron(8)](https://man7.org/linux/man-pages/man8/anacron.8.html)). A portable application-owned equivalent gives the same semantic on launchd, systemd, or cron: every trigger calls a due-check, and the due-check plus idempotent sweep supplies catch-up. Unlike replaying every missed tick, one current-state sweep is normally sufficient for append-only source trees.

Blind spot: a hybrid still fails if every trigger path is broken or the sweep falsely reports success. Independent `doctor` checks must inspect both OS trigger state and application coverage.

## Reliability plumbing

### Overlap prevention and atomic progress

launchd and systemd timers serialize the manager's own instance, but they do not serialize unrelated manual/hook invocations. Put a nonblocking per-user lock around the shared sync operation. Linux can use `flock(2)`/`flock(1)` on an opened file; macOS does not ship the `flock` CLI, so use an advisory lock in the implementation or an atomic lock directory with PID/start-time validation. A contending nudge should set a durable dirty flag or exit successfully knowing the active runner will sweep again. Never treat “lock busy” as proof that work will be covered unless that handoff exists.

Write progress with temp-file + fsync where warranted + atomic rename. Useful state is: schema/version; run ID; started/finished timestamps; trigger reason; source roots examined; counts/bytes; highest source mtime or provider cursor observed; success/failure and error summary; archive reconciliation result; and tool version. Update `last-success` only after every required root has been enumerated and outputs committed. Keep `last-attempt` separate so repeated failures remain visible.

### Failure backoff

A periodic job naturally waits until the next timer tick, but transient failures may justify bounded retries with exponential backoff and jitter. Cap attempts and total time so a broken path does not become a forever crash loop. A daemon should let launchd/systemd respawn it, using `ThrottleInterval` or `RestartSec` plus systemd `StartLimitIntervalSec`/`StartLimitBurst` ([systemd.service](https://www.freedesktop.org/software/systemd/man/latest/systemd.service.html), [systemd.unit](https://www.freedesktop.org/software/systemd/man/latest/systemd.unit.html)). Authentication or permissions failures are not helped by tight retries; record them and preserve the next scheduled sweep.

### Logs for a forever job

On Linux/systemd, prefer stdout/stderr to journald and let `SystemMaxUse`, retention, and rotation policy bound storage ([journald.conf](https://www.freedesktop.org/software/systemd/man/latest/journald.conf.html)). On macOS, either use unified logging from native code or own a bounded log policy; launchd's `StandardOutPath` and `StandardErrorPath` append/create files but do not provide application log rotation. If plain files are required, rotate by size with a small fixed generation count under the same lock. Cron needs explicit redirection plus logrotate/newsyslog integration or a bounded application log. The durable last-run record should remain structured and separate from human logs.

### What `doctor` can verify per OS

“Live schedule” and “complete archive” are separate propositions. The OS can establish registration, enablement, next/last trigger, process state, and recent exit. It cannot know whether Packbat enumerated every source or produced complete output. That requires application state and an independent reconciliation scan.

| Platform / mechanism | Prove registration and liveness | Prove recent execution | Prove no work is missed | Important limit |
|---|---|---|---|---|
| macOS launchd agent | Validate the installed plist and exact paths; `launchctl print gui/$UID/<label>` must find the job and expose its current diagnostic state/config; for a daemon, verify a live PID plus a fresh heartbeat. `launchctl print-disabled gui/$UID` can expose an override that disables the label. | Inspect currently emitted `launchctl print` fields such as run count and last exit status where present, then require a fresh application `last-attempt` and successful `last-success` stamp. Optionally `kickstart` a probe and wait for a new run ID. | Independently enumerate every configured source root and compare each eligible file/version against archive metadata; verify the sweep's recorded roots and cursor/high-water mark. Calendar wake catch-up plus `RunAtLoad` is policy evidence, not coverage evidence. | Apple marks `launchctl print` output as **not API**, so a doctor must tolerate OS-version changes and treat it as diagnostic evidence, not a stable parser contract. launchd has no durable history of every scheduled firing; a loaded one-shot is normally “not running,” and power-off misses are invisible unless the app detects overdue work. See [`launchctl` source/manual](https://github.com/apple-oss-distributions/launchd/blob/main/man/launchctl.1). |
| Linux systemd user timer | `systemctl --user show packbat-sync.timer` and service: require `LoadState=loaded`, `UnitFileState=enabled`, expected `ActiveState`/`SubState`, correct `FragmentPath`, and expected timer properties. `systemctl --user list-timers --all`/`show` exposes next and last trigger. If no-login execution is required, `loginctl show-user "$USER" -p Linger` must say yes. | Timer `LastTriggerUSec`; service `Result`, `ExecMainStatus`, `ExecMainCode`, timestamps, and journal; then require fresh Packbat attempt/success stamps. A probe can `systemctl --user start packbat-sync.service` and wait for a new run ID. | Reconcile source against archive. Verify `Persistent=yes` and a durable timer timestamp for catch-up, but rely on Packbat's last-success/high-water state for successful coverage. | Timer “last trigger” can point to a failed service. `Persistent` coalesces missed periods to one activation and only works when the timer is activated. [`systemctl show`](https://www.freedesktop.org/software/systemd/man/latest/systemctl.html), [`systemd.timer`](https://www.freedesktop.org/software/systemd/man/latest/systemd.timer.html). |
| cron fallback | `crontab -l` must contain exactly one wizard-owned, correctly parsed entry; verify executable/config paths and that the cron daemon is available/running using distro-specific service inspection. | Cron has no portable per-job status. Only Packbat's fresh attempt/success stamps and logs prove invocation; a doctor-initiated probe proves the command works now. | Reconcile every source file/version. Detect overdue `last-success` at startup/doctor because cron supplies no catch-up. | A correct line can still never run due to daemon, power, environment, DST, permissions, or path problems. No portable next-run or last-exit query. |
| Watcher daemon under launchd/systemd | Manager state/PID, fresh heartbeat, expected watched-root count, and “watch backend healthy” status; Linux additionally compare watch usage/errors against inotify limits. | Fresh event-loop heartbeat and last-event/last-sweep stamps; inject a harmless canary in a dedicated watched test directory and require a new processed event/run ID. | Periodic independent full reconciliation; require zero unresolved overflow/dropped-event/root-replaced flags and a sweep after the most recent such flag. | A PID and heartbeat do not prove every directory is watched; a canary proves one path, not the whole tree. |

A strong `doctor` therefore reports at least four independently colored facts: **installed** (owned artifact matches expected content), **scheduler/daemon live**, **last successful sweep fresh**, and **coverage reconciled now**. “No work missed” is only defensible after the fourth check. A lightweight default doctor can sample mtimes/cursors; `doctor --deep` can enumerate and compare all source files.

## How real tools do it

**Syncthing.** Syncthing is a long-lived continuous-sync daemon. Its folder model enables an OS filesystem watcher for low latency, accumulates watcher events for `fsWatcherDelayS`, and still runs a full scan every `rescanIntervalS`; the rescan interval can be zero only when something external guarantees scans. This is the canonical watcher-plus-periodic-reconciliation hybrid: notifications improve latency, the scan repairs loss, and daemon/service management is separate from sync correctness ([Syncthing configuration](https://docs.syncthing.net/users/config.html), [Syncthing source](https://github.com/syncthing/syncthing)).

**resticprofile and autorestic.** resticprofile owns a genuine cross-platform schedule lifecycle: profile config becomes OS-native timer artifacts, and its commands install, remove, and show platform-specific status; on systemd it writes units then reloads/enables/starts them. Autorestic instead stores per-location cron expressions and expects an external frequent `autorestic cron` poller; its docs tell the user to edit crontab and show a line with a fixed PATH. The former is a scheduler-unit wizard; the latter does not install a schedule and is an application due-check behind a coarse scheduler ([resticprofile schedule commands](https://creativeprojects.github.io/resticprofile/schedules/commands/), [autorestic cron docs](https://git.nicco.io/cupcakearmy/autorestic/src/commit/fc2756124e74ac20a2b4fe3dc2545f329a43f28c/docs/markdown/location/cron.md)).

**Time Machine.** Time Machine presents automatic periodic backup rather than a user-visible cron job: Apple documents hourly, daily, and weekly backups, hourly APFS local snapshots when the destination is absent, automatic schedule resumption when a destination reconnects, and Power Nap backups on supported Macs connected to power. Its lesson is policy-level catch-up around changing availability, with local staging/snapshots and destination-aware resumption, not merely firing at an exact wall-clock instant ([Apple Time Machine overview](https://support.apple.com/guide/mac-help/back-up-your-files-mh35860/mac), [backup frequency and reconnect](https://support.apple.com/en-us/104984), [Power Nap](https://support.apple.com/guide/mac-help/turn-power-nap-on-or-off-mh40774/mac)).

**SpecStory watcher.** SpecStory's terminal CLI offers `specstory watch` for new supported-agent activity and auto-exports transcripts into `.specstory/history`; `specstory sync` separately retroactively exports every available session from providers' stored histories. That public contract is itself a hybrid: watch for low latency, explicit sweep/sync for history and recovery. Its docs describe provider source locations, local project state, SQLite session index, optional logs, and cloud/local outputs, but do not publicly promise OS autostart, watcher event durability, or missed-event recovery, so those properties should not be inferred ([terminal-agent overview](https://docs.specstory.com/integrations/terminal-coding-agents), [CLI usage and state](https://docs.specstory.com/integrations/terminal-coding-agents/usage)).

**Git auto-sync tools.** Git-oriented auto-sync projects commonly run a loop or periodic scheduler that detects a dirty worktree, then add/commit/pull/push; an example is `GitJournal`, which makes Git synchronization part of an application, while small “repo watcher” projects poll upstream on an interval ([GitJournal source](https://github.com/GitJournal/GitJournal), [git-repo-watcher source](https://github.com/kolbasa/git-repo-watcher)). Their key analogy is negative: an observed clean worktree or successful push is inspectable, but concurrent edits, conflicts, authentication, non-fast-forward history, and ignored/untracked files expand the state machine far beyond “event happened.” For Packbat's one-way append-only copy, retain the useful parts: serialize, keep a durable last-success, poll/sweep to reconcile, and expose conflicts/errors rather than equating a live loop with coverage.

## Failure-mode matrix

| Pattern | Failure it is naturally blind to |
|---|---|
| Calendar/interval scheduler | A successfully launched sweep that skipped a root/file; corrupt or stale output; exact missed-firing history; configuration paths that later disappear. |
| launchd calendar | Power-off calendar instants; disabled override; no-login period for a LaunchAgent; output log growth. |
| systemd persistent timer | Multiple missed occurrences (coalesced); failures after trigger; user manager absent without linger; timer enabled but never activated. |
| cron | Sleep/off misses; overlap; per-job exit state; runtime environment mismatch; DST edge behavior. |
| FSEvents watcher | Dropped/coalesced events until flags are handled; invalid old event IDs; root/volume replacement; unreadable paths. |
| inotify watcher | Recursive setup races; watch/queue exhaustion; daemon downtime; unsupported/network filesystem behavior. |
| Harness/shell hook | Crashes, SIGKILL, power loss, sleep before completion, disabled hooks, unsupported/new harnesses, writes after hook. |
| Daemon supervisor | Live but wedged process; healthy process with incomplete watch set; repeated respawn without useful work. |
| Any stamp-only doctor | False-success stamps, clock jumps, roots omitted from configuration, source mutations newer than the stamp. |

## How the dogfood maps onto the landscape

The dogfood script is the reconciliation half of a hybrid even though it currently has only scheduler triggers. Its full-tree, mtime-based, overwrite-newer sweep is idempotent enough for calendar catch-up and `RunAtLoad`; hostname scoping avoids cross-machine writers; append-only output prevents deletion propagation. The launchd plist uses the stronger macOS calendar trigger, so sleep misses catch up once, and `RunAtLoad` catches many login/reboot cases. It does not catch a 21:00 power-off miss until the next load or next 21:00.

The gaps are observability rather than archive-format choices: the script's final “complete” line is not conditioned on all commands succeeding; stderr and completion share one unrotated file; the fixed `/opt/homebrew/bin/zstd` path can break after architecture/package changes; there is no explicit cross-trigger lock; no structured last-attempt/last-success record; and no source-to-archive reconciliation. A `doctor` can prove the launch agent is loaded and read launchd's last exit, but cannot presently prove that every discovered JSONL has a current `.zst`. These observations describe the baseline; they are not a product decision.

## Open questions (12)

1. What is the maximum acceptable archive lag during normal use and after a laptop was powered off for days?
2. Must Linux sync run before first login/after logout, or is login-triggered persistent catch-up sufficient? This determines whether enabling linger is necessary.
3. Which Linux environments are supported when systemd user managers are absent: cron only, a portable daemon autostart path, or an explicit unsupported case?
4. Is `doctor` allowed to perform a full O(number of session files) reconciliation by default, or should deep proof be an explicit slower mode?
5. What source-side version proves a growing JSONL is current: mtime/size, content hash, provider cursor/session ID, or a stored byte offset?
6. Can hooks be installed into each supported harness without taking ownership of or overwriting user hook configuration, and what merge/uninstall contract is required?
7. Does the daemon/watch option justify its resident memory, battery, packaging, and native-backend complexity compared with hook nudges plus frequent sweeps?
8. What macOS privacy permissions can a wizard reliably obtain or diagnose for all agent session roots, especially when the executable path changes after upgrade?
9. Should a contending nudge persist a dirty flag for an immediate follow-up, or is the next scheduled reconciliation within the lag target?
10. What retention and privacy policy applies to operational logs and structured run/error records?
11. Which clock should freshness use across suspend and wall-clock changes: wall time for user display plus monotonic duration within a boot, or provider/source high-water marks alone?
12. What exact evidence threshold lets `doctor` use the word “healthy” versus “installed,” “running,” “overdue,” or “coverage unverified”?
