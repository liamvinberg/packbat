# Encryption for Packbat's off-box leg

Research current to 2026-07-12. The recommendation assumes that the off-box copy is a disaster-recovery copy of Packbat's existing archive, not a second canonical archive format.

## Recommendation

1. **Use age per archive file, then send the ciphertext tree with rclone or a native transport.** Keep only the age recipient, which is public, on the machine during normal operation. Give the user the identity once as a recovery kit and do not require it to run backups. This is the cleanest resolution of “a key only you hold” and “you never think about it”: the secret decryption key is not needed for unattended encryption at all. age has a small, specified format, authenticated streaming encryption, independent interoperable Go and Rust implementations, and no repository state or locks. The cost is no cross-file deduplication and a full re-encryption plus full-object upload whenever an active compressed archive file is replaced.
2. **Use restic if measured active-file churn makes full uploads unacceptable, or if built-in repository verification is valued above preserving the ordinary ciphertext tree.** Restic is a mature, one-binary backup engine with content-defined chunking, interruption-safe snapshots, integrity checking, and native S3/SFTP/B2 support. Its symmetric repository password must be available on the source machine for zero-interaction backups, so Packbat must store it in an OS keychain or a protected file and separately force a recovery-copy moment.
3. **rclone crypt** is the simplest one-binary encrypted mirror and has the broadest remote support, but it combines transport and encryption configuration, keeps a recoverable symmetric secret on the source machine, and reuploads a changed file in full.
4. **Kopia** is technically credible and is the only candidate here with built-in OS-credential-store integration, but its repository maintenance and ownership rules are more machinery than this append-mostly copy needs.
5. **A private Git remote** is familiar but is a poor backup substrate for regenerated ciphertext. Every changed age-encrypted file is a new incompressible blob, history grows permanently, filenames and commit metadata leak, and hosted Git limits are aimed away from generated archives.

This ranking changes if remote filename confidentiality is mandatory and packbat is unwilling to add a stable opaque-name mapping plus encrypted manifest. age encrypts file contents, not pathnames; restic and rclone crypt encrypt repository metadata or names as part of their formats.

## Option notes

### age and rage

age is unusually well matched to Packbat's trust invariant because encryption and decryption do not use the same secret. The installed backup job can retain a public `age1…` recipient indefinitely without gaining the ability to decrypt its own remote backup. This is stronger and simpler than protecting an automation password in a keychain.

The format is seekable and streaming, and each file is independently recoverable. A corrupt object does not poison an entire repository. That independence also means there is no global inventory, snapshot transaction, deduplication, or missing-object check. Packbat or its transport must publish uploads atomically and retain enough authenticated inventory to distinguish an intentionally absent file from remote loss.

Use the Go `age` CLI as the default compatibility target. `rage` is a credible drop-in implementation and useful recovery fallback, but selecting it as the bundled default gains little: both have prebuilt binaries, while age is the reference implementation and has stable 1.x releases. Do not reuse an SSH identity as the archive identity. age's own documentation warns that SSH keys are commonly treated as revocable authentication keys rather than long-lived decryption keys, and SSH-agent-held keys are not supported ([SSH key caveats](https://github.com/FiloSottile/age#ssh-keys)).

### restic

Restic's decisive advantage is repository-level work. It detects changes, chunks content before encryption, deduplicates locally, uploads immutable packs, commits a snapshot, and can later verify that the whole snapshot is restorable. Re-running an interrupted backup uploads only what is still needed ([resume FAQ](https://restic.readthedocs.io/en/latest/faq.html#will-restic-resume-an-interrupted-backup)). This is the best behavior for a frequently replaced large file if its compressed bytes remain substantially stable.

The trade is conceptual and operational. The off-box result is a restic repository rather than Packbat's encrypted tree, and recovery depends on repository metadata and restic's format. Locks are normally harmless but add a state users can encounter. Restic also assumes snapshots may later be forgotten and packs pruned; Packbat would need an explicit no-forget/no-prune policy to preserve its never-delete invariant. The repository password is needed for every backup, so public-key-only automation is impossible.

### rclone crypt

rclone crypt preserves a browsable one-object-per-file shape and encrypts filenames by default. That makes it more turnkey than composing age with rclone, and it supports nearly any remote the user is likely to own. Its weakness is key custody: the lightly obscured config is effectively the key, while encrypting that config introduces another password that automation must retrieve. This can be made acceptable with a keychain wrapper, but rclone does not provide the clean public-only steady state that age does.

Its integrity model is per file/block, not per backup set. `rclone copy`, rather than `sync`, fits the never-delete invariant because it copies changed/new files without deleting destination-only files ([copy semantics](https://rclone.org/commands/rclone_copy/)). Replacing an active file overwrites the logical destination object; remote bucket versioning is required if older active versions must survive off-box.

### Private Git remote

Plain Git over SSH with files pre-encrypted by age is safer than git-aware transparent encryption, but it still mismatches the workload. Git's strengths are textual deltas and human history. age ciphertext defeats those deltas, while every historical ciphertext remains reachable. GitHub's own guidance says programmatically generated files belong in object storage, not Git.

`git-crypt` is explicitly not fit. Its maintainers say it is best for a few secret files in an otherwise public repository, not most or all files. It does not encrypt names or metadata, cannot revoke access/rotate keys across history, and every small plaintext change produces an entire new incompressible ciphertext ([documented limitations](https://github.com/AGWA/git-crypt#limitations)). It is GPL-3.0 and adds GPG or exported-key UX.

`transcrypt` has the same category problem: Git clean/smudge filters protect selected secrets inside a working repository. It is not a backup repository, does not remove Git metadata leakage, and makes correctness depend on filters being installed and applied. Neither tool should be in the shortlist.

### Kopia

Kopia materially beats restic on one UX axis: it stores the repository password in macOS Keychain, Windows Credential Manager, or a Linux keyring by default ([CLI credential storage](https://kopia.io/docs/reference/command-line/)). It also offers deduplication, authenticated encryption, native S3/B2/SFTP, and full-data verification. Default encryption is AES-256-GCM-HMAC-SHA-256, with ChaCha20-Poly1305 available ([FAQ](https://kopia.io/docs/faqs/)).

It does not displace restic here. Kopia has automatic quick/full maintenance, a designated maintenance owner, lock and clock-skew assumptions, and garbage-collection behavior ([maintenance model](https://kopia.io/docs/advanced/maintenance/)). Its rclone backend is experimental. The built-in keychain integration is worth copying at the wrapper layer, not enough reason to adopt the more complex repository.

## What good turnkey products do

Three patterns recur in strong peer UX:

- **Hide the daily secret retrieval behind the OS.** Restic officially documents using `RESTIC_PASSWORD_COMMAND` to call macOS Keychain and warns against exposing secrets in process arguments/environments ([automation FAQ](https://restic.readthedocs.io/en/latest/faq.html#how-can-i-specify-encryption-passwords-automatically)). Kopia makes OS credential storage the default rather than wrapper glue.
- **Generate a high-entropy secret instead of asking the user to invent one.** age and rage generate a secure multi-word passphrase when passphrase mode is selected ([age passphrases](https://github.com/FiloSottile/age#passphrases)). For Packbat's recommended recipient mode, a random identity is stronger and does not need to be memorized.
- **Create one explicit recovery ceremony.** 1Password automatically generates a 128-bit Secret Key, keeps working copies in device keychains/backups, and puts the recovery material in a printable [Emergency Kit](https://support.1password.com/emergency-kit/). Tailscale Tailnet Lock displays disablement secrets only during initialization and explicitly asks users to store them in a password manager or print them; losing all copies makes the locked tailnet unrecoverable ([Tailnet Lock onboarding](https://tailscale.com/docs/features/tailnet-lock#disablement-secrets)). Tailscale's optional vendor escrow is not compatible with Packbat's invariant, but its show-once warning and multiple physical storage suggestions are.

Restic wrappers such as resticprofile improve scheduling and configuration, but they generally forward `password-file`/`password-command` rather than solving recovery. The useful peer lesson is therefore not another wrapper dependency. It is a deliberate split between automatic local retrieval and a user-controlled disaster copy.

## Proposed key UX

The following proposal is for the recommended age path.

### 1. Generation

During the off-box wizard, generate a dedicated native age X25519 identity with a CSPRNG and derive its public recipient. Never reuse an SSH key. Configure the remote and complete a small encrypt, upload, download, and decrypt smoke test before declaring setup successful.

### 2. Routine storage

Persist only the public recipient in Packbat's local configuration. It is not secret. During encryption, write each ciphertext to a same-filesystem temporary path, close and verify success, then atomically rename it into the outbound tree before upload. Store remote access credentials using the remote adapter's normal secure mechanism; they are authorization credentials, not the archive decryption key.

The age identity should not remain in Packbat's normal config or OS keychain after onboarding. Keeping it in a password manager is fine because that is the user's chosen custody, not a Packbat service. If product testing finds that forcing immediate off-device custody causes excessive abandonment, the fallback is OS keychain storage plus a prominent “this laptop copy is not your recovery copy” status, but that weakens the clean public-only design.

### 3. Backup-the-key moment

Before the wizard can finish, create a small text recovery kit containing:

- the age identity;
- its public recipient/fingerprint for verification;
- the remote type and non-secret locator, such as endpoint, bucket and prefix or SSH host/path;
- exact fresh-machine restore commands and the Packbat version/format marker;
- one sentence in plain language: **If every copy of this identity is lost, nobody can recover this archive.**

Offer two concrete destinations: save the kit to a user-chosen path for a password manager/secure drive, or print it to the terminal for immediate capture. Do not upload it to the backup remote and do not offer packbat/vendor escrow. Confirm custody by asking the user to re-enter a short checksum/fingerprint from the saved kit, not the secret itself. Then remove any temporary identity file and report that future backups need no key prompt.

This is intentionally one moment of friction. Making the wizard dismissible without a verified recovery copy would satisfy zero-interaction backup while failing the laptop-loss invariant the feature exists to address.

### 4. Fresh-machine restore

`packbat restore` should ask for or discover the recovery kit, configure/authenticate the remote separately, download an authenticated inventory plus requested ciphertexts, verify the kit recipient matches the archive, and decrypt into a new destination. It should never overwrite an existing local archive implicitly. The user needs three independent things: remote locator, remote access, and the age identity. Loss of remote access is an account/provider problem; loss of the identity is cryptographically irreversible; loss/corruption of remote objects is a backup-integrity failure.

For the restic runner-up, use the same ceremony but generate a random high-entropy repository password, store the daily copy through `RESTIC_PASSWORD_COMMAND` in macOS Keychain/Linux Secret Service where available, put the password in the recovery kit, and add a second restic key only if it represents a genuinely independent recovery copy. On headless Linux without a usable secret service, a mode-0600 password file is the practical fallback, but it makes the off-device recovery kit mandatory.

## Open questions

1. **Must remote pathnames be confidential?** age does not encrypt names. If yes, decide and specify the stable opaque-name/encrypted-inventory scheme before choosing age; otherwise restic or rclone crypt wins this axis out of the box.
2. **How large and how frequently replaced is an active `.zst` file?** Benchmark age/rclone full replacement against restic on real append-and-recompress traces. Restic's CDC advantage on compressed bytes is plausible but not guaranteed.
3. **Must old versions of the active session file survive remotely, or only its newest replacement?** The local invariant says “re-replaced” and “never deleted,” which is ambiguous about the superseded version. The answer affects object versioning and restic snapshot retention.
4. **Is one transport binary acceptable?** age plus rclone is two downloaded executables; age plus narrow native S3/SFTP adapters reduces binary weight but expands Packbat's own transport maintenance surface.
5. **Which Linux environments count as first-class?** Desktop Secret Service, headless servers, containers, and WSL have different keyring availability. This mostly affects the restic/rclone fallbacks, not age's public-only steady state.
6. **What authenticated inventory owns missing-file detection?** age authenticates present files but cannot prove that a complete remote tree was returned. Define whether Packbat, the transport, or provider versioning/immutability supplies set-level integrity.
7. **What is the binary update trust policy?** Choose bundled-per-platform npm packages versus pinned first-use downloads, and specify checksum/signature verification, caching, updates, and offline restore media.
8. **Is post-quantum recipient support a requirement now?** age 1.3 supports hybrid post-quantum recipients, but their public recipient strings are much larger and rage compatibility/version floors must be pinned. X25519 is the simpler default unless the threat model explicitly requires harvest-now-decrypt-later protection.

## Confidence

**High** on the relative mechanics and the public-key UX advantage. **Medium-high** on the final ranking because two product facts remain decisive and unmeasured: filename-confidentiality requirements and real bandwidth behavior of replaced zstd files. Audit confidence is **medium**: official specifications, designs, release/security records, and interoperability are available, but no candidate advertises a completed independent cryptographic audit.
